MVP — Token Engine & Builder
Mês 2–4
▾
Arquitetura de tokens: semânticos vs componente
Duas abordagens possíveis. Semânticos (spacing.positive.comfortable) são mais flexíveis e reusáveis entre componentes. Tokens de componente (button.background.primary) são mais intuitivos pro editor visual e pra quem monta componentes custom no dashboard. Pode ser híbrido: camada semântica como base + mapeamento automático pra tokens de componente gerados pelo editor.
TECH DECISÃO CRÍTICA
🔴 Debate necessário — Token Architecture
Opção A — Semânticos only: Mais puro, mais reusável, menos acoplado. Mas o editor visual precisa resolver o mapeamento "qual semântico vai pra qual prop do componente" — complexidade no dashboard.

Opção B — Component tokens only: Mais intuitivo pro editor, 1:1 com props. Mas gera explosão de tokens e dificulta consistência cross-component.

Opção C — Híbrido (recomendada): Semânticos como source of truth. O editor visual gera component tokens automaticamente como alias dos semânticos. O consumidor final vê tokens de componente, mas por baixo é tudo semântico. Melhor dos dois mundos, mais complexo de implementar.

→ Precisa de spike técnico antes de decidir.
Token compiler (novo, do zero)
Motor de compilação que recebe a definição de tokens e gera os outputs. W3C DTCG spec como formato de entrada. Outputs: CSS custom properties, SCSS variables, JSON, JCSS. Multi-brand e light/dark resolvidos aqui via data-attributes no CSS — sem runtime provider.
TECH DEFINIDO
Token Builder — Figma plugin vs Web app
Designers montam a folha de tokens visualmente. Exporta no formato correto pro compiler. Duas opções de superfície, podem coexistir.
TECH NOVO
🔴 Debate necessário — Figma Plugin vs Web
Figma Plugin: Designer não sai do Figma, flow natural. Mas: Figma plugin API é limitada pra UIs complexas (iframe com 0 acesso a APIs do browser, UI toolkit restrito), plugins existentes (Tokens Studio) já são ruins — fazer melhor é possível mas custoso. Manutenção alta porque depende da API do Figma que muda.

Web App: Controle total da UX, tech stack familiar (React), pode ser muito mais poderosa. Mas: designer precisa sair do Figma, é mais uma ferramenta no workflow.

Sugestão: Web app primeiro (mais rápido de iterar, melhor UX, sem dependência de API externa). Figma plugin como fase futura — sync bidirecional leve (push tokens pro Figma, pull Figma variables pro builder) sem tentar ser o editor principal dentro do Figma.

→ Precisa investigar: quão viável é um plugin Figma decente? Limitações atuais da API?
CLI para consumo local
@dsbuilder/cli — puxa os tokens compilados pro projeto, roda o build local. Primeiro ponto de contato do dev com o produto. Precisa ser dead simple: npx dsbuilder init → escolhe tema → tokens no projeto.

2
Componentes Core & Kit Open Source
Mês 4–7
▾
Biblioteca styled com tokens integrados
Button, Modal, Card, Input, Select, Tooltip, Accordion, Tabs, etc. Componentes styled (não headless) — já vêm com visual funcional usando os tokens do builder. Override via props e CSS custom properties. Compound component patterns com prefixed naming (CardHeader, não Card.Header).
TECH DEFINIDO NOVO
Props standardization
Padrão de props consistente: layout (size, spacing), behavior (disabled, loading), variant (primary, secondary), mode (light, dark). Tipagem forte com TypeScript. Cada componente documenta suas props no mesmo formato.
TECH DEFINIDO
Kit npm open source
@dsbuilder/core publicado no npm. Tree-shakeable (tsup), SSR-friendly (Next.js App Router compatível), zero runtime pra temas (CSS only). Porta de entrada do ecossistema — atrai devs, gera stars, valida o produto.
TECH DEFINIDO
Storybook + documentação
Storybook react-vite como playground dos componentes. Cada componente com stories interativas, docs de props, e exemplos de uso. Serve tanto como documentação quanto como ferramenta de QA.
TECH DEFINIDO
Acessibilidade como baseline
WCAG AA como mínimo. ARIA attributes, keyboard navigation, focus management. Diferencial competitivo: muitos DS builders ignoram a11y. Se os componentes core já vêm acessíveis, é selling point forte.
TECH NEGÓCIO
3
Editor Visual — Produto Pago
Mês 7–10
▾
Editor visual de componentes custom
Interface web para construir componentes visualmente. Output é código real com tags HTML semânticas (section, article, nav, header, footer — não só divs). User friendly, preview em tempo real, exporta componente pronto pro kit. Esta é a feature premium — precisa ser excelente.
TECH NEGÓCIO MAIS COMPLEXO
🔴 Maior desafio técnico do projeto
O editor visual é essencialmente um "Figma pra componentes React com HTML semântico". Isso é extremamente ambicioso. Pontos a resolver:

1. Modelo de construção: Drag-and-drop de elementos HTML? Árvore de nós editável? Builder baseado em blocos (como Notion)? Cada approach tem trade-offs de complexidade vs usabilidade.

2. HTML semântico: Como garantir que o output usa tags corretas? O editor precisa sugerir/forçar semântica? Ou o usuário escolhe a tag de cada elemento?

3. Binding de tokens: Como o usuário conecta tokens aos estilos do componente? Visual picker? Dropdown? Autocomplete?

4. Output de código: O componente gerado é JSX puro? Usa algum runtime? Como manter o código limpo e legível?

5. Responsividade: O editor precisa suportar breakpoints? Como o usuário define o comportamento responsivo?

→ Recomendo: spike técnico de 2-3 semanas focado SÓ no editor antes de começar a fase 3. Prototipar o modelo de construção com 1 componente simples (Button) e 1 composto (Card).
Multi-brand & temas via compilação
Troca de marca/tema resolvida no build time via compilação de CSS. Sem ThemeProvider, sem runtime overhead. Data-attributes no HTML root controlam o tema. Cada brand gera seu próprio set de CSS custom properties. SSR-friendly por design.
TECH DEFINIDO NOVO
Versionamento & changelog
Cada alteração no DS gera uma versão. Consumidores fazem pin em versão específica. Changesets-style workflow.
TECH
Infra cloud do dashboard
Onde roda a app do editor? Como armazena os projetos dos usuários? CDN pra servir os assets compilados? Vercel/Cloudflare pra app, S3/R2 pra assets, Postgres pra projetos. Custos iniciais estimados: ~$50-100/mês.
